从零打造「全自动无人驾驶」Hexo 博客：独立爬虫定时抓取与云端 Actions 联动避坑全纪录

你是否也曾想过让自己的 Hexo 博客实现真正的“无人驾驶”？比如每天定时自动抓取全网最新的优质资讯（如机场节点、VPN 测试等）并自动发布，同时自己平时在本地电脑写博客时，互不干扰，完全不需要考虑同步和冲突问题？

这两天我亲自踩通了这条流水线，经历了 “网页全白”、“403 权限拒绝”、“本地主题锁死” 等一连串天坑。最终，成功实现了独立爬虫定时运行 ➔ 自动推送源码 ➔ 云端全自动编译发布，与本地手动 hexo d 完美共存的闭环。

今天就把这套真正的双端无感解耦方案的搭建全过程，以及血淋淋的避坑自救指南完整记录下来！

一、 核心设计：双端无感解耦方案

很多教程在设计自动化爬虫时，会逼着用户每次在本地写博客前都要同步云端数据，极其麻烦。

而我们这套方案的核心精妙之处在于：彻底解耦，互不干扰。 * 云端：爬虫定时运行，把抓取到的 daily.md 弹射给源码仓库，并由云端 Actions 编译发布到部署分支。

  • 本地：你平时开机写你自己的文章，写完直接执行经典的 hexo g -d 部署。

因为本地部署和云端部署最终都是将静态网页推送到同一个部署分支（如 gh-pages），两端的 Markdown 源码层级彻底隔离。你本地不需要运行任何爬虫，不需要配 Token，更不需要天天 git pull 搞同步，真正做到了双端并行、纯粹的无人驾驶！

三、 第二阶段：那些让我们彻夜难眠的「连环天坑」与自救

当流水线跑起来后，真正的挑战在于云端环境和本地环境的微小差异。以下是我亲自踩过的两个核心大坑：

💥 天坑 1：博客云端部署 Actions 报错 403 (Permission Denied)

【现象】：云端爬虫成功把 daily.md 送进了源码仓库，也自动触发了源码仓库的 Hexo 编译。但在最后一步将打包好的网页往 gh-pages 线上分支推送时突然中断，报错 Permission denied to github-actions[bot]，博客直接崩溃变白。

【病因】：GitHub 默认系统自带的 GITHUB_TOKEN 只有读取权限，没有写入权限。云端打包机有权编译，但无权发布。
【解药】：打开博客源码仓库网页端，点击 Settings -> Actions -> General，页面下拉找到 Workflow permissions，将默认选项改选为 “Read and write permissions”（读和写权限）并保存。

💥 天坑 2：云端显示部署成功（绿色对勾），但网页为什么依旧是一片空白？

【现象】：权限给足后，Actions 一路绿灯。但刷新博客，网页依然是一片空白，查看日志发现所有的 CSS 样式文件和 JS 脚本都没生成。

**/病因（双重冲突）/`**：

  1. 云端打包机缺少样式渲染插件：GitHub Actions 容器只运行了基础的 npm install，而默认环境缺少类似 hexo-renderer-stylus 这样的样式渲染插件，导致生成出来的全是裸奔的 HTML 空壳。
  2. 本地主题被误判为独立的子模块（Submodule）：我发现本地的 themes/landscape 目录下残留了一个名为 .git (或写成 git) 的隐藏文件夹。这导致本地在往云端推代码时，Git 直接跳过了整个主题的样式和模板文件！云端源码仓库里其实只有一个空壳主题。

四、 终极胜利

两端配置彻底调顺的那一瞬间，多米诺骨牌完美咬合！云端打包机（Actions）仅用 19秒 完成完美编译，GitHub Pages 发布机器人仅用 23秒 无缝上架。

现在，这套系统已经坚如磐石：云端爬虫每天雷打不动地抓取并更新资讯，而我自己在本地开机就能写随笔，写完一个 hexo d 直接发布，双端并行，互不干扰，这才叫真正的“全自动无人驾驶”！
