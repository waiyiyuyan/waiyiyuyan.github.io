从零打造「全自动无人驾驶」Hexo 博客：独立爬虫定时抓取与 GitHub Actions 联动避坑全纪录

你是否也曾想过让自己的 Hexo 博客实现“无人驾驶”？比如每天定时自动抓取全网最新的优质资讯（如机场节点、VPN 测试等），自动生成博文并发布，同时还不影响自己平时在本地手动写稿？

这两天我亲自踩通了这条流水线，经历了 “网页全白”、“403 权限拒绝”、“本地 Git 锁死推送” 等一连串世界级天坑。最终，成功实现了独立爬虫定时运行 ➔ 自动推送源码 ➔ 云端全自动打包编译 ➔ 线上无缝发布的完美闭环。

今天就把这套双仓库物理隔离方案的搭建全过程，以及这两天血淋淋的避坑自救指南完整记录下来，分享给所有折腾 Hexo 的朋友！

一、 为什么选择「双仓库物理隔离」方案？

如果把爬虫脚本直接放在 Hexo 博客的根目录下，让云端 Actions 每天定时改动代码并就地打包，看似简单，却隐藏着一个致命痛点：

Git 状态污染。 云端每天都在背着你修改文件，而你本地电脑对这一切消极不知情。当你哪天在本地写完一篇文章准备 git push 时，就会无情地迎来大红色的 [rejected] main -> main (fetch first) 推送拒绝，天天都在跟云端解冲突，极其痛苦。

为了彻底解放双手，我们采用双仓库物理隔离方案：

  1. 博客源码仓库（waiyiyuyan.github.io）：只存放 Hexo 博客的源码（Markdown 原始文件、主题、配置等），保护你本地的心血不被污染。
  2. 独立爬虫仓库（ygpy-crawler）：完全运行在云端，专门负责每天定时起床运行爬虫脚本，生成单个 daily.md 文件，并像精准导弹一样利用 Git 远程发射到博客源码仓库中。

本地电脑不再需要运行爬虫，不配任何高危 Token。云端爬虫与本地创作互不干扰，通过标准 Git 流水线自动完成数据对齐与云端打包编译。以下是两仓库与 GitHub Actions 联动运行的全局拓扑图：

三、 第二阶段：那些让我们彻夜难眠的「连环天坑」与自救

当爬虫仓库高高兴兴把 daily.md 弹射到了博客源码仓库后，真正的“恶魔试炼”才刚刚开始。以下是我这两天亲身踩过的、最典型的三个大坑与实战自救经历：

💥 连环坑 1：博客部署 Actions 报错 403 (Permission Denied)

【现象】：第一晚两边仓库刚连通，爬虫成功把 daily.md 送了过来。源码仓库也检测到代码变了，高高兴兴启动了 Hexo Auto Deploy。但是在最后一步要将编译好的网页推送到 gh-pages 分支发布时，突然中断死掉了，报错 Permission denied to github-actions[bot]。随后，整个博客网站直接崩溃成了一片空白。

【病因】：GitHub 出于安全保护考虑，默认情况下系统自带的 GITHUB_TOKEN 只有读取权限，没有写入权限。打包机有权编译，但无权发布，被 GitHub 的安保系统当场拦截。

【解药】：

  1. 打开你的博客源码仓库网页端，点击右上角的 **Settings**。
  2. 左侧菜单栏找到 Actions -> 点击 **General**。
  3. 页面下拉找到 **Workflow permissions**。
  4. 将默认的 “Read repository contents…” 改选为 “Read and write permissions”（读和写权限），保存即可！到这一步，写权限的问题就被完美干掉了。

💥 连环坑 2：权限通了（显示绿色对勾），但网页为什么依旧是一片空白？

【现象】：开启写权限后，第二天中午 12:49 爬虫再度运行，这一次 Actions 一路绿灯，给出了完美的绿色对勾（✓）！但刷新博客前台，网页竟然依然是一片死寂的空白，看日志发现所有的 CSS 样式文件和 JS 脚本都没生成。

【病因（双重背锅侠）】：

  1. 云端打包机没有安装样式渲染器：本地运行 Hexo 依赖我们电脑里装好的各种 npm 插件，但 GitHub Actions 开辟的干净容器只运行了最基础的 npm install，它根本不知道你的主题需要特殊的渲染插件（如 hexo-renderer-stylus 等）。
  2. Git Submodule（子模块）幽灵封锁：检查本地项目时，我发现 themes/ 主题根目录下居然残留了一个名为 .git (或写成 git) 的旧隐藏文件夹。在 Git 看来，这意味着 themes/landscape 主题是一个“外部独立的独立子仓库”。这就导致你在本地执行 git add . 时，Git 直接把整个主题核心模板和样式文件给跳过了！ 云端仓库里只有一个空壳子，编译出来自然没有样式。

💥 连环坑 3：为了修复配置，本地运行 git push 被无情拦截拒绝？

【现象】：今天下午，当我改好本地的 deploy.yml 强补渲染器配置，在终端满怀希望地运行 git push 准备拯救博客时，Git 突然弹起大红色的无情拒绝：! [rejected] main -> main (fetch first)。

【病因】：这不是因为云端权限不够（Actions 还没触发），而是因为中午 12:49 时，云端的定时自动爬虫已经成功运行，并且往云端的 main 分支里塞入了包含最新标题的全新的 daily.md。此时云端的进度已经悄悄走在了我本地电脑的前面。Git 为了保护云端好不容易抓下来的文章不被我本地缺失该文件的状态覆盖，启动了保护拦截。

【解药 —— 日常手动开发的“黄金守则”】：
